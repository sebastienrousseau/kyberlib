# Architecture

> Last Updated: 2026-05-22

`kyberlib-wasm` is the WebAssembly binding crate for
`kyberlib`. It exposes a small JS-callable surface
(`keypair`, `encapsulate`, `decapsulate`) via
`wasm-bindgen` so a browser or Deno / Bun runtime can use
FIPS 203 ML-KEM directly.

For the per-item API reference see
[docs.rs/kyberlib-wasm](https://docs.rs/kyberlib-wasm); for
*using* the crate from JS / TS see [`usage.md`](./usage.md).

## Why a sidecar crate

In `kyberlib` v0.0.6 a `wasm` Cargo feature on the core
crate pulled in `wasm-bindgen` and friends. v0.0.7 split
the WASM surface into this sidecar because:

1. **The safe core stays dependency-light.** `kyberlib` is
   `#![no_std]` under default features with no proc-macro
   dependencies. Pulling `wasm-bindgen` into the core
   would have added 14 transitive crates to every build.
2. **The `wasm` feature was a no-op on non-wasm targets**
   anyway — every consumer who didn't want WASM had to
   carry the `[features]` line. Cleaner to make WASM an
   *external* dependency.
3. **Bundle size.** A WASM-targeting consumer pulls
   exactly the JS-bindable surface (one `*.wasm` blob +
   `wasm-bindgen` JS glue), not the full crate. With
   `wasm-opt -O3` the resulting binary is ~120 KiB
   compressed.

The relocation was tracked by [#144]; it landed in v0.0.7.

## Surface

The crate exports four JS-callable functions and two
opaque types:

```rust,ignore
// JS-callable functions:
pub fn keypair() -> Result<Keys, JsError>;
pub fn encapsulate(pk: Box<[u8]>) -> Result<Kex, JsValue>;
pub fn decapsulate(ct: Box<[u8]>, sk: Box<[u8]>) -> Result<Box<[u8]>, JsError>;

// JS-visible types:
pub struct Keys { /* pubkey, secret */ }
pub struct Kex  { /* ciphertext, shared_secret */ }
```

These mirror the legacy free-function API in `kyberlib`
(byte-slice in, byte-slice out). The typed v0.0.7 API
(`MlKem768::generate` etc.) is **not** exposed via WASM —
JS doesn't usefully consume Rust's typed-state model, and
exposing the raw byte API matches what existing JS / TS
KEM libraries (`@noble/post-quantum`, `@cloudflare/pq`)
expose.

## Parameter set

The default parameter set is **ML-KEM-768** (matching
`kyberlib`'s default). Compiling against ML-KEM-512 or
ML-KEM-1024 requires:

```sh
cargo build --target wasm32-unknown-unknown \
    --no-default-features \
    --features kyber512   # or kyber1024
```

The byte lengths in the JS surface change accordingly —
see [`usage.md`](./usage.md) for the per-parameter table.

A future enhancement (issue
[#180](https://github.com/sebastienrousseau/kyberlib/issues/180))
will expose all three parameter sets in a single WASM
binary by routing through `kyberlib`'s v0.0.7 typed API.
This is held back because adding three sets to the WASM
surface roughly doubles the bundle size, which most
consumers don't need.

## Unsafe code

The crate is **not** `#![forbid(unsafe_code)]` because
`wasm-bindgen`'s glue code uses `unsafe` to marshal
between JS and Rust types. The wrapping ratio is favourable
though — only `wasm-bindgen`'s internals contain `unsafe`,
not anything in `kyberlib-wasm/src/`.

The crate is `#![deny(missing_docs)]` so every public
function carries rustdoc.

## RNG sourcing

The WASM build uses `rand::rngs::OsRng`. On the wasm32
target this routes via `getrandom` 0.2+ to the browser's
`crypto.getRandomValues` (or Node.js's `crypto.randomBytes`
under Node-WASI). Both surfaces are seeded from the host
OS's CSPRNG.

This means: on the WASM build, you do *not* need to plug
in an explicit RNG — the `keypair()` function takes no
args and consults the platform CSPRNG internally. (On the
native build, ergonomics prefer accepting an `&mut impl
RngCore + CryptoRng` — see the bare `kyberlib` crate.)

## Bundle anatomy

A release `wasm-pack build --release` produces:

```text
pkg/
├── kyberlib_wasm.js            JS glue (~7 KiB)
├── kyberlib_wasm_bg.wasm       compiled WASM (~120 KiB after wasm-opt -O3)
├── kyberlib_wasm.d.ts          TypeScript declarations (~3 KiB)
├── package.json                npm metadata
└── README.md
```

The `*.wasm` file dominates; the JS glue and TS types are
auto-generated by `wasm-bindgen`. wasm-opt is run as part
of `wasm-pack`'s release pipeline.

## Bundle size budget

The release-gate budget is 150 KiB compressed
(`brotli -q 11`) for the `.wasm` blob. Current measurement
on v0.0.7: ~122 KiB. Headroom: ~28 KiB before the gate
fails.

## Targets

* `wasm32-unknown-unknown` — the primary target. Used in
  browsers, Deno, Bun, and Cloudflare Workers.
* `wasm32-wasip2` — works in WASI runtimes. Not the
  primary target but the build passes.
* `wasm32-unknown-emscripten` — **not supported**. Use
  `wasm-bindgen` instead.

## What this crate does NOT do

* **Streaming encryption** — ML-KEM is a KEM, not a
  cipher. Use the 32-byte shared secret with a symmetric
  AEAD (`@noble/ciphers`, `WebCrypto`'s AES-GCM).
* **PEM / PKCS#8 encoding in WASM** — out of scope; that
  belongs in `kyberlib-pkcs8` and a future WASM sidecar
  for it.
* **Hybrid construction in WASM** — `kyberlib-hybrid` does
  not currently ship a WASM binding. Tracked as
  [#181](https://github.com/sebastienrousseau/kyberlib/issues/181).
* **TypedArray ergonomics** — the byte-slice surface uses
  `Box<[u8]>`. JS callers get `Uint8Array` for free via
  wasm-bindgen, but the conversion is one-way (Uint8Array
  → Vec<u8>); the JS side copies, so don't expect zero-
  copy.
